> Исполнитель: ИИ-агент (Claude Code или аналог). Человек утверждает деплои
> в прод, платные операции и коммиты. План составлен 2026-07-09.

## 0. Контекст (прочитать перед началом)

Два репозитория:

| Репо | Путь | Что это |
|---|---|---|
| `bodia2010/exam-trainer` | `/home/igor/project/exam_trainer` | Flutter-приложение (Android основная цель) |
| `bodia2010/exam-trainer-api` | `/home/igor/project/exam-trainer-api` | Python/Flask на Vercel: MarkItDown-конвертация PDF, Gemini-парсинг, Firestore REST, Upstash Redis |

Ключевые факты:
- Аутентификация — Firebase ID token в каждом запросе (`Authorization: Bearer`), общего секрета нет.
- Два Gemini-ключа: `GEMINI_API_KEY` (премиум) и `GEMINI_API_KEY_FREE` (бесплатный тариф, свой биллинг).
- Все 12 типов разделов + discovery защищены `responseSchema` (`exam-trainer-api/response_schemas.py`); генерация конфига — единственный источник истины `generation_config.py`.
- Кеш: Upstash через `/api/cache`, версия кеша зашита в клиент (`parse_service.dart` → `_cacheVersion`, сейчас v13). **Любое изменение промпта/схемы = поднять версию.**
- promptfoo-евалы: `exam-trainer-api/promptfoo/` (14 parse-тестов + 2 discovery). Discovery-regression-тест известно флаки (~1 из 3), это задокументировано в yaml.
- Незакоммиченная работа: в обоих репо есть большой объём незакоммиченных изменений. Git push в main блокируется в auto-режиме — спрашивать пользователя.

Правила для агента:
- Перед деплоем в прод (`vercel --prod`) — всегда явно спросить пользователя.
- После изменения промптов/схем: поднять `_cacheVersion`, прогнать promptfoo, пересобрать APK.
- Сборка APK: `flutter build apk --debug --dart-define=API_BASE_URL=https://exam-trainer-api.vercel.app`.
- Не коммитить без запроса пользователя.
- Free-ключ Gemini доступен для локальных тестов промптов (тот, что в Vercel env `GEMINI_API_KEY_FREE`); не светить его в выводе/ошибках.

## Фаза 1. Наблюдаемость (сначала — иначе оптимизируем вслепую)

### 1.1 Логирование реального расхода токенов
**Задача:** в `main.py::_call_gemini` читать `usageMetadata` из ответа Gemini
(`promptTokenCount`, `candidatesTokenCount`, `thoughtsTokenCount`) и писать в лог
(`print` → Vercel logs) с меткой `section_type` и тарифом (free/premium).
**Зачем:** спор «что дороже — discovery или parse» сейчас решается на пальцах.
**Приёмка:** после одного тестового импорта в `vercel logs` видна строка на каждый
Gemini-вызов с точными цифрами. Ничего не падает, если `usageMetadata` отсутствует.

### 1.2 Метрика cache hit/miss
**Задача:** логировать в `/api/cache` (GET) попадание/промах с типом ключа
(doc / group / discover — различимы по префиксу до хеширования: добавить
префикс в сам ключ на клиенте, напр. `v13|group|<hash>`; учесть — это
инвалидация кеша, совместить с ближайшим бампом версии).
**Приёмка:** по логам за день видно соотношение hit/miss по каждому типу.

### 1.3 Сводка стоимости импорта
**Задача:** после данных из 1.1 — скрипт `exam-trainer-api/scripts/cost_report.py`,
агрегирующий логи в таблицу: средний импорт = X input + Y output токенов = $Z
(тарифы gemini-3.1-flash-lite: $0.25/$1.50 за 1M in/out).
**Приёмка:** одна команда выдаёт отчёт; числа согласуются с биллингом Google.

## Фаза 2. Оптимизация стоимости и скорости

### 2.1 Ревизия thinkingLevel для heavy-типов (после 1.1!)
**Гипотеза:** после внедрения responseSchema повышенный `thinkingLevel: LOW` для
`_HEAVY_SECTION_TYPES` может быть избыточен — схема уже гарантирует полноту.
**Шаги:** добавить в promptfoo провайдер-вариант с MINIMAL для heavy-типов,
прогнать 3 раза все 14 parse-тестов. Если 100% стабильно — понизить в
`generation_config.py`, замерить экономию по данным 1.1.
**Приёмка:** решение (оставить/понизить) с числами в комментарии кода.

### 2.2 Сжатие discovery-входа
**Проблема:** discovery шлёт весь документ (~150K токенов) на каждый новый файл.
**Проверить гипотезы (по очереди, промером на реальном фикстуре
`promptfoo/fixtures/discover_input.txt`):**
a) страницы-оглавления/ссылки можно предварительно вырезать эвристикой без ИИ;
b) пустые строки схлопнуть (несколько подряд → одна) — дёшево, безопасно;
c) НЕ резать документ на куски — уже решили, что риск границ выше выгоды.
**Приёмка:** discovery-евал проходит на сжатом входе с тем же item count (±2),
экономия input-токенов задокументирована.

### 2.3 TTL для rate-limit ключей и мусора в Upstash
**Задача:** проверить в Upstash-консоли объём хранимого; у cache-ключей нет TTL
(намеренно — «parsed content never changes»), но старые версии (v1..v12) мёртвые.
Написать одноразовый скрипт очистки ключей со старыми префиксами версий
(безопасно: они больше никогда не будут прочитаны).
**Приёмка:** объём базы уменьшился; текущие v13-ключи не тронуты.

### 2.4 Размер APK
**Факт:** debug-APK ~184 МБ — это норма для debug, но release не собирался ни разу.
**Шаги:** `flutter build apk --release --dart-define=API_BASE_URL=...` (потребует
keystore — если его нет, остановиться и спросить пользователя), затем
`flutter build appbundle --release --analyze-size`. Проверить, что
`cached_network_image`/`audioplayers`/шрифты не тянут лишнего.
**Приёмка:** release-APK собирается; размер и топ-5 тяжёлых компонентов — в отчёте.

## Фаза 3. Корректность и тесты

### 3.1 Первые widget-тесты (сейчас тестов ноль)
Использовать установленный скилл `flutter-add-widget-test`. Приоритет по риску:
1. `parse_service.dart`: `_validateGroup`/`_validateShape` — чистые функции,
   тестируются без моков (подать битые/валидные JSON-объекты каждого типа);
2. `_expandSentinels` и `_mergeByVariant` — исторически багоопасные места;
3. `course_storage.dart`: изоляция по UID (смена мок-юзера → другой список);
4. `favorites_service.dart`: `removeByCourse` каскад.
**Приёмка:** `flutter test` зелёный, покрыты перечисленные функции; тесты не
ходят в сеть и в Firebase (интерфейсы замокать минимально).

### 3.2 Стабилизация promptfoo как регресс-гейта
**Задача:** скрипт `promptfoo/run_all.sh`: оба конфига, discovery — 3 прогона
(из-за известной флакучести), выход ненулевой только при стабильном провале.
**Приёмка:** одна команда = полный вердикт; README обновлён.

### 3.3 E2E-проверка импорта (руками агента через backend)
**Задача:** скрипт `exam-trainer-api/scripts/e2e_import.py`: берёт PDF (путь
передаёт пользователь), проходит convert → discover → parse всех найденных
типов **в обоих тарифах** (нужны два тестовых аккаунта — токены передаёт
пользователь), валидирует результат теми же правилами, что клиент.
**Приёмка:** скрипт печатает по-секционную таблицу OK/FAIL; на эталонном PDF
всё OK; free-тариф возвращает ровно 1 вариант на раздел.

### 3.4 Кросс-девайс синхронизация курсов
**Проверить:** импорт под аккаунтом A → `/api/courses` (GET) возвращает курс;
удаление курса → курс исчезает из Firestore; курс > 900KB — пропускается без
ошибки импорта (найти/сделать большой PDF или замокать).
**Приёмка:** чек-лист из 3 пунктов с фактическими результатами.

## Фаза 4. Безопасность

### 4.1 Аудит Firestore security rules
Использовать установленный скилл `firebase-security-rules-auditor`.
**Контекст:** клиент Firestore напрямую НЕ читает (всё через backend с service
account), но правила всё равно должны запрещать прямой доступ — проверить, что
в консоли Firebase не остались открытые правила по умолчанию.
**Приёмка:** правила = deny all для клиентов (или явное обоснование исключений).

### 4.2 Проверка утечек в ответах API
**Задача:** grep-аудит `main.py` на все `jsonify({'error': str(e)})` — убедиться,
что ни один путь не может отдать наружу ключ/токен/внутренний URL (прецедент
уже был с `raise_for_status`). Особо: `convert()` и `tts_endpoint()` всё ещё
отдают сырой `str(e)`.
**Приёмка:** сырые исключения наружу не уходят ни в одном эндпоинте.

### 4.3 Лимиты бесплатного тарифа
**Проверить обходы:** отправка >12K символов / >8 редакций на `/api/parse`
free-токеном → 403; `discover` free-токеном с документом >N — сейчас НЕ
ограничен по размеру вообще (это осознанно? зафиксировать решение);
rate-limit 1000/час срабатывает (можно проверить скриптом с меньшим порогом
на preview-деплое, не в проде).
**Приёмка:** таблица «атака → фактический ответ сервера».

### 4.4 Секреты в репозиториях
`git log -p | grep`-аудит обоих репо на исторические утечки (ключи, токены,
service account JSON). Внимание: гитхаб-токен уже светится в remote URL
deutch-lernen — проверить, не попал ли он в файлы exam-trainer.
**Приёмка:** список находок; если найдено — план ротации (выполнять только
после подтверждения пользователя).

## Фаза 5. Подготовка к релизу (Play Store)

### 5.1 Функция удаления аккаунта — **блокер публикации**
Google Play требует in-app удаление аккаунта + веб-форму запроса. Сейчас нет
ни того, ни другого. Реализация: пункт в шите аккаунта → подтверждение →
backend-эндпоинт `/api/account` (DELETE): удалить Firestore-данные
(users/{uid} + подколлекции devices/courses/progress), затем Firebase Auth
пользователя (REST `accounts:delete`), клиент: локальные данные UID + signOut.
Веб-форма — статичная страница на GitHub Pages рядом с privacy.html.
**Приёмка:** аккаунт удаляется end-to-end; повторный вход создаёт чистый аккаунт.

### 5.2 Release-подпись и версия
Keystore (спросить пользователя — есть ли), `key.properties` вне git,
`versionCode`/`versionName` в pubspec, минификация включена по умолчанию.
**Приёмка:** подписанный appbundle собирается воспроизводимо.

### 5.3 Чек-лист Play Console (заполняет пользователь, агент готовит тексты)
Privacy policy URL (уже есть: bodia2010.github.io/exam-trainer/privacy.html),
Data safety форма (список собираемых данных — взять из privacy policy),
скриншоты (агент может подготовить через эмулятор), описание на 4 языках
(тексты из strings.dart как основа).
**Приёмка:** документ `PLAY_SUBMISSION.md` со всеми текстами и статусами.

## Порядок выполнения

```
1 (наблюдаемость) → 2 (оптимизация по данным) 
                  ↘ 3 (тесты) — можно параллельно с 2
4 (безопасность) — независимо, можно в любой момент
5 (релиз) — последней, 5.1 можно начать сразу (блокер с длинным хвостом)
```

Каждая фаза: отдельная ветка → показ диффа пользователю → его решение о
коммите/деплое. Не смешивать фазы в одном диффе.
